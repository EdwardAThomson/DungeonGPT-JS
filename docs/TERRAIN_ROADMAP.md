# Terrain Roadmap

Consolidated plan for evolving the world map terrain system. Distinguishes what is actually shipped today from what has been designed but not yet integrated into the production game.

---

## 1. Current State

The production world map is generated by `src/utils/mapGenerator.js` via `generateMapData(width, height, seed, customNames)`. It is a seeded, rule-based placement pass — **not** a noise/heightmap pipeline. Tiles start as `plains`, then a single edge is converted to a water/beach coast (`placeCoast`), followed by 1–2 lake clusters, 3–5 forest clusters, 2–3 mountain ranges, 2–4 towns (`placeTown`), rivers routed from mountains to water via A* pathfinding (`pathfinding.js`), and a post-pass that fills sparse quadrants (`improveMapDistribution`). Mountain tiles are then flood-filled into named clusters (`harmonizeMountainNames`). The result is a 10x10 grid of tiles with `biome`, `poi`, `townName`/`townSize`, `mountainName`, `hasRiver`, `beachDirection`, etc.

Rendering lives in `src/components/WorldMapDisplay.js`. Each cell is a 40x40px div with a flat biome background color (`#c3e6cb` plains, `#007bff` water, `#f5deb3` beach) and a Unicode emoji overlay for POIs (`🌲` forest, `⛰️` mountain, `🕳️` cave, `🛖🏡🏘️🏰` towns by size). Roads and rivers are SVG paths (`M20,0 L20,40`-style) stroked in brown or royal blue, composited on top of the biome color. There are no image assets for terrain tiles; the visual is entirely CSS colors plus emoji plus inline SVG.

**Experimental / debug-only work exists but is not wired into gameplay.** `src/experimental/mapGen/layeredGenerator.js` implements a multi-layer noise pipeline and is consumed by `src/pages/TerrainStudioV2.js` (route `/debug/terrain-studio-v2`) with a 3D preview via `src/experimental/components/TerrainMesh3D.js`. An older `TerrainStudio.js` + `organicGenerator.js` pair lives alongside it. None of these feed `NewGame.js`, `useGameMap.js`, or `WorldMapDisplay.js` — the production path still calls `generateMapData` and renders emoji.

## 2. Roadmap: Layered Terrain Generation

Replace the rule-based placement generator with a noise-driven heightmap that produces natural overland terrain with sharp peaks and wide valleys. A prototype already exists at `src/experimental/mapGen/layeredGenerator.js`; the work is to harden it and swap it into the production path.

**Pipeline (5 layers):**

1. **Multi-octave Simplex/Perlin noise** — terrain skeleton. Parameters: `octaves` (4–8), `persistence` (0.4–0.6), `lacunarity` (~2.0), `baseFreq` (0.005–0.02). Standard FBM accumulation, normalised to [-1, 1].
2. **Continental mask** — a separate very-low-frequency noise field (`freq` 0.001–0.005) smoothstepped around `seaLevel`, multiplied against layer 1 to carve land vs water.
3. **Domain warping** — sample layer 1 at coordinates displaced by two decorrelated noise fields. `warpStrength` 0.5–2.0, `warpFreq` 0.005–0.015. This is the single biggest contributor to "looks natural" vs "looks like Perlin blobs".
4. **Power redistribution** — `pow(normalised, exponent)` with exponent 1.5–2.0. Cheaply produces sharp peaks with broad valleys without paying for DLA.
5. **Hydraulic erosion (optional)** — particle-based droplet simulation that carves river valleys and deposits sediment. 50k–200k particles, one-time post-process cost (~200ms–2s depending on grid size).

**Recommended defaults:** `octaves: 6, persistence: 0.5, lacunarity: 2.0, baseFreq: 0.01, continent.freq: 0.003, warp.strength: 1.2, warp.freq: 0.008, exponent: 1.8, erosion.particles: 100000`.

**Performance targets:** layers 1–4 on 512x512 ≈ 20–80ms (real-time slider preview). Adding erosion pushes to 200–500ms; acceptable as a one-time generation cost.

**Integration work required to ship:**

- Decide the grid resolution for production (current emoji map is 10x10; a heightmap-driven version likely wants 64+ internally, with biome classification quantised to the display grid).
- Map heightmap bands to the existing biome vocabulary (`plains`, `water`, `beach`, plus `forest`/`mountain` POIs) so downstream code (pathfinding, milestone spawner, town placement) keeps working.
- Port town/river/road placement onto the new heightmap (towns on low-slope land, rivers following gradient descent instead of the current A*-to-nearest-water).
- Keep `customNames` injection intact for milestone-driven town/mountain naming.

## 3. Roadmap: AI Image Tiles

Replace emoji + flat-color rendering with AI-generated top-down tile images. The rendering layer (`WorldMapDisplay.js`) is small and would swap each emoji/CSS background for an `<img>` or CSS `background-image`.

**Tile inventory (~33 tiles total):**

| Category | Count | Difficulty |
|---|---|---|
| Base terrain (plains, deep water, shallow water, beach) | 4 | Easy — standalone |
| Features (forest, mountain, cave entrance) | 3 | Easy — standalone |
| Towns (hamlet, village, town, city) | 4 | Easy — standalone |
| Coast transitions (4 straight + 4 corners) | 4–8 | Hard — must align with neighbours |
| Road overlays (N-S, E-W, 4 corners, intersection) | 7 | Medium — need transparency |
| River overlays (same 7 directions) | 7 | Medium — need transparency |

**Prompt template:** `Top-down seamless [subject] tile, fantasy RPG style, [details], square tile, game asset, seamless edges`. Include phrases like `fills the entire frame`, `edge-to-edge`, `no padding` to fight the standard AI-generator whitespace problem. Midjourney's `--tile` flag or Stable Diffusion tiling workflows are preferred for anything that must wrap.

**Post-processing:** generate at 512x512 or 1024x1024, auto-trim whitespace (`convert input -trim -resize 128x128!`), downscale to 128x128 for the game (current render is 40x40 CSS px; higher source res gives headroom for zoom).

**Coast alignment — the hard problem.** Options in order of risk:

- **A. Keep CSS gradients for coasts** — only replace standalone terrain/feature/town tiles. Low risk, still big visual upgrade.
- **B. Directional tiles with fixed edge colors** — prompt constrains water/land edge hex exactly to the neighbour's base color so seams line up.
- **C. CSS `mask-image` blending** — AI tiles + CSS gradient masks at coast boundaries.
- **D. Full tileset generation** — one specialised workflow produces the whole matching set at once.

**Phased rollout:**

1. **Phase 1 (low risk):** Features + towns + cave, 7 tiles. No edge-matching needed.
2. **Phase 2 (medium risk):** Base terrain textures (plains, deep water, shallow water, beach). Keep CSS coasts (Option A).
3. **Phase 3 (high risk):** Coast transitions, 4–8 tiles. Only if Phase 2 coasts look bad.
4. **Phase 4 (optional):** Road/river overlay tiles. Current SVG strokes may already look fine over textured tiles — evaluate first.

**2D vs 3D:** an experimental 3D system already exists (`WorldMapDisplay3D.js`, `TerrainMesh3D.js`) using solid colors. Good 2D tiles would double as textures if the 3D path is revived, so start 2D.

## 4. Roadmap: Terrain Studio Controls

UI spec for the parameter panel that drives the layered generator. Already partially implemented in `src/pages/TerrainStudioV2.js`.

**Global**
- **Seed** (number): deterministic world selector. Same seed = same map.
- **Resolution**: 64 / 128 / 256 / 512. Use 128–256 while experimenting, 512 for final render (~1–2s).

**Noise Layers**
- **Octaves (2–8):** detail stack. 2–3 smooth hills; 5–6 natural; 7–8 busy texture.
- **Persistence / Roughness (0.3–0.7):** per-octave amplitude falloff. 0.3–0.4 smooth, 0.5 default, 0.6–0.7 jagged.
- **Sea Level (0–100%):** continental mask threshold. Low = mostly land, high = archipelago.

**Shaping**
- **Domain Warp Strength (0–2.5):** the "naturalness" dial — the single most impactful control. 0 = Perlin blobs, 1.0–1.5 organic default, 2.0+ extreme.
- **Peak Exponent (1.0–3.0):** peak-to-valley ratio. 1.0 linear, 1.3–1.8 natural mountain ranges, 2.0+ spiky alpine, 3.0+ near-flat with needle peaks.

**Erosion**
- **Hydraulic Erosion (toggle):** simulate droplets flowing downhill to carve river valleys.
- **Particles (10k–200k):** 10k–30k subtle smoothing; 50k–80k visible valleys; 100k+ deep channels. Works best at resolution 256+.

**Recommended presets:**

| Style | Octaves | Persistence | Sea Level | Warp | Exponent |
|---|---|---|---|---|---|
| Rolling countryside | 4–5 | 0.4 | 20% | 0.5 | 1.0 |
| Mountainous island | 6–7 | 0.5 | 40% | 1.2 | 1.5 |
| Dramatic alpine | 8 | 0.45 | 30% | 1.5 | 2.0 |
| Archipelago | 5 | 0.5 | 60% | 1.0 | 1.3 |

**Camera (3D preview):** click+drag / Q-E rotate, right-drag / WASD pan, scroll zoom, `2D MAP` button toggles top-down.

## 5. Status & Decisions

Emoji tiles remain in production because the current rule-based generator + CSS-color + emoji stack is cheap, deterministic, mobile-friendly, and tightly coupled to downstream systems (milestone spawner injects named POIs into `customNames`; pathfinding consumes biome enums; town generation triggers on tile enter). The layered generator exists only as a 3D debug tool at `/debug/terrain-studio-v2` with no biome classification, no POI placement, and no integration into `useGameMap.js` or `NewGame.js`. To move forward, the work is (1) add a heightmap-to-biome quantiser so `generateLayeredTerrain` can emit the same tile schema `generateMapData` does, (2) port town/river/road placement onto the new heightmap, (3) decide 2D image tiles vs revived 3D as the target renderer, and (4) plan a Phase-1 AI-tile swap for standalone tiles (features + towns) that doesn't require coast edge-matching. Until steps 1–2 land, the emoji map stays.
